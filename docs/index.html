说说Redis中的持久化

说说Redis中的持久化 基本介绍 首先对于redis来说，假如其宕机了那就需要恢复，假如说从数据库恢复数据出来，有可能redis中的数据众多，恢复成本较大，造成数据库访问压力增大，另外一个情况就是数据库的文件是存储在磁盘当中的，恢复速度比较的慢，redis的响应速度也会跟着变慢 基于数据恢复的场景，redis有着两大机制实现持久化：AOF和RDB AOF AOF日志的写入和redolog这种WAL日志不同，AOF是一种写后日志，先执行命令再进行日志的记录，这样的一个好处就是写入日志的命令一定是有效的，不需要进行检查，因为其已经经过了执行，还有一个好处就是不会阻塞当前的写操作 写回策略 AOF这种写后日志有优点的同时也有着缺点，一是假如redis再执行命令成功后写入AOF日志的过程中宕机了，那么这一段数据就消失了，对于将redis作为数据库的系统是风险很大的，二是写入AOF日志是由主线程负责的，假如这个过程阻塞了，那么对于后续的请求主线程也无法处理 这些缺点实际上也是和AOF写回磁盘的时机有关，redis对于这种情况有三种写回磁盘的策略 同步写回：每一个命令执行成功后立即将日志写回磁盘 每秒写回：每一个命令执行成功后立即将日志写入AOF文件的内存缓冲区当中，每一秒将缓冲区内容写入磁盘 OS写回：每一个命令执行成功后立即将日志写入AOF文件的内存缓冲区当中，由操作系统自行决定什么时候将缓冲区内容写回磁盘 日志重写 当然随着命令在AOF文件中的逐渐增加，AOF日志文件也会跟着变的越来越大，这就需要对其文件大小进行一定的限制，通常这个过程叫做AOF日志的重写 AOF日志通常记录的是一条一条的命令，那么有一些命令就是对相同的键值对进行修改，那么我们只需要记录最新一个命令即可并将其替换之前的一些命令，这样的话，在数据恢复的时候只需要执行这一条命令即可。不过要注意的是，虽然AOF日志重写后大小变小，但是重写的过程也是一个非常耗时的操作 AOF日志的重写由后台子进程bgrewriteof来操作的，这也是为了避免阻塞主线程。AOF日志重写的过程可以总结为：一个拷贝，两处日志 一个拷贝指的就是就是每次执行重写的时候，主线程fork出的bgrewriteof子进程会把主线程的内存拷贝给子进程，这里面就包含了数据库的最新数据。然后子进程就可以在不影响主线程的情况下将拷贝的数据写成操作记入重写日志 两处日志指的就是因为是子进程处理日志重写，主线程依旧可以处理新来的请求。假如这个时候有写操作，第一处日志指的就是正在使用的AOF日志，也就是原来的AOF日志，redis会将新的写操作记录在这个缓冲区当中，这样即使宕机了，这个日志依旧是完整的 第二处日志就是指新的AOF重写日志，这写操作也会被写到重写日志的缓冲区当中，这样操作的话，即使新的写操作在重写日志中也不会丢失。等所有拷贝数据的所有操作记录完成重写之后，重写日志中这些记录后来的写操作也会被写入新的AOF重写日志，以保证数据库最新状态。这个时候就算完成重写，之后就可以用新的AOF重写日志代替旧的日志 RDB RDB就是另外一种redis持久化方式，也就是**内存快照。**内存快照就是将内存中的数据在某个时刻的状态记录，也就是类似于照片记录效果的方式，将某一时刻的状态以文件的形式将其写到磁盘上，也就是快照。这样一来，即使宕机，快照文件也不会丢失，数据的可靠性也得到了保证。这个快照文件也被叫做RDB（Redis DataBase）文件 和AOF日志相比，RDB记录的是某一个时刻的数据而并不是操作，所以对于数据恢复的时候，直接将RDB文件读入内存，可以很快的完成数据恢复，但同时也有着一定的顾虑也就是： 对哪一些数据执行快照，这关系到快照的执行效率问题 做快照时，数据是否能增删改，这关系到主线程是否阻塞 对于这种顾虑，因为数据都在内存当中，redis是直接采用了全量快照的方式，将内存中的所有数据都记录到磁盘当中，同时对于RDB文件的生成，有两种方式： save：在主线程中执行，会导致阻塞 bgsave：创建一个子进程，专门用于写入RDB文件，避免主线程阻塞，这也是RDB方式的默认配置 写时复制 当然对于在对内存数据做快照的时候，redis采用的方式是写时复制的方法进行内存快照的，具体而言，就是在主线程进行写操作的时候，采用os提供的Copy-On-Write来进行写操作 简单的来说，bgsave子进程是由主线程fork出来的，可以共享主线程的所有内存数据，在bgsave子进程开始运行之后，开始读取主线程的内存数据，并把其写入至RDB文件中 对于写操作的键值对，主线程直接生成键值对的一个副本，对副本进行操作，而bgsave子进程则直接对原来的内存数据进行快照生成RDB文件 快照频率 对于快照频率的话，假如在进行快照的过程中redis又进行了写操作，而正好这时宕机了，那么这段时间写操作的数据就丢失了，所以把握快照时间的间隔也非常的关键 假如快照的频率非常的高呢？比如每秒做一次快照。那么对于这种情况其实也有一定问题的，每次进行后台快照都是由**bgsave子进程进行的，**那么频繁的快照也会造成磁盘io压力的增大，同时频繁的创建子进程会造成主线程的阻塞 对于这种情况采取的办法就是进行增量快照，简单来说就是在做了一次全量快照之后，后续的快照只对修改过的数据进行快照记录，这样可以避免每次全量快照的开销 在第一次做完全量快照之后，只需要将被修改的数据写入RDB文件即可，不过要注意的是哪些是被修改的数据，有些情况下记录这些被修改的数据本身空间占用就非常大，所以这也是一个问题 对于redis4.0之后，采用了一种混合AOF日志和内存快照的方式进行持久化。简单的来说，就是内存快照以一定的频率执行，在两次快照之间，使用AOF日志记录这期间的所有命令操作，这样做的好处就是，内存快照的频率可以缩减，也就是总体上fork子进程带来的资源消耗变少，AOF也只要记录两次内存快照之间的操作，这样可以避免代价过大的重写开销 一个场景问题 使用一台2C4G，500g的主机运行redis，redis数据库大小差不多为2g，使用RDB方式做持久化处理，假如此机器redis的处理请求方面写读比例差不多8:2左右，那么分析下在这个场景下RDB持久化的风险 首先是内存风险：主线程fork子进程进行bgsave，那么写的比例是80%，意味着大概有1.6g的空间用于副本进行存储主线程的写操作加起来就相当于总和3.6g，假如父进程此时有大量key进行写入，那么内存将很快用完，假如没有swap的话就会直接oom，导致进程被os关掉，开启swap的话导致redis读数据的性能急剧下降 其次是cpu风险，由于是2c的，主线程用于处理请求，bgsave用于RDB快照操作，而这个过程是大量消耗cpu资源的，但同时redis进程还有很多其他线程在后台操作，比如AOF每秒刷盘，异步关闭fd等，那么此时的RDB持久化可能就会演变成cpu资源之间的竞争，这么做的结果就会导致主线程处理请求的延迟加大，RDB处理时间也会加大，整体性能下降

Redis中的基本数据类型

Redis中的基本数据类型 基本介绍 redis包含五种键值对值的数据类型，分别是：**String，List，Hash，Sorted Set，Set。 **对于键值对中的键，都是String类型的，因为redis是k-v数据库，所以整体而言数据的保存形式就相当于在一张大的哈希表中存储，值存储各种数据类型 同时对于字符串而言，只有字符串的底层数据结构是只有一种SDS，其他的都包含两种数据结构，主要体现为在不同情况下转换底层的数据结构以提高存储效率 键和值的结构组织方式 对于redis而言，为了实现从键到值的快速访问，redis使用了一张全局哈希表来保存所有的键值对 哈希表其实就是一个数组，数组中的每一个元素都称为哈希桶，每个哈希桶都保存了键值对数据，哈希桶中的元素保存的实际上是指向具体值的指针 哈希冲突 对于越来越多的键值对的存储，特定长度的数组在哈希存储时必然会出现哈希冲突，即多个entry同时映射到同一个数组索引上，对于这种现象，redis采用了链式哈希的办法来将entry同时存储在同一个数组索引上面，即后面加入的entry接在前面的entry上，他们之间以链表的形式进行链接 当然越来越多的entry在同一个哈希表索引上时，对于链上靠后的entry的查询时间复杂度就会上升，这时候redis就会重新对所有的entry进行rehash扩容，类比于java中的hashmap，当然hashmap还会对链表进行转红黑树操作，这就有所不同了 对于rehash而言，redis默认采用了两个全局哈希表，我们将之命名为哈希表1和哈希表2。 当一开始插入数据时，默认使用哈希表1，当数据开始增多之后，redis开始进行rehash操作，主要分为三部操作： 为哈希表2分配更大的空间，通常是2倍，以容纳哈希表1rehash后的entry数 把哈希表1中的数据重新映射rehash并拷贝到哈希表2当中 释放哈希表1的空间 对于哈希表1而言，作为下一次rehash时中的to区 渐进式rehash 当然对于上面那种rehash而言，假如enrty数量过多的话，rehash的时长必然非常的长，造成无法响应其他请求，对于这种情况redis采用了渐进式rehash的策略 具体渐进式rehash的操作步骤主要是将对enrty的rehash计算分摊至每个请求当中，比如说在处理第一个请求的时候，同时将哈希表1中索引为0的全部entry进行rehash，处理第二个请求的时候将索引为1的全部entry进行rehash，这样就不必一次全部rehash了，同时也能够响应客户端的请求了 进行rehash的出发条件就是负载因子，就是哈希表已保存节点数量/哈希表大小 底层数据结构 SDS 对于redis中的string数据类型，底层是简单动态字符串，其相比较于c原生的char数组而言，自封装了一个结构体用于承载字符串，具体表现为 对于c原生字符串，SDS的好处就是能够直接获取字符串长度，直接进行预分配，不用担心是否数据溢出，同时其保存的数据是二进制安全的，可以存储包含"\0"的数据，当对字符串进行修改时也是安全的，其会先计算预留的空间，假如不足直接扩容，充足则直接分配追加 双向链表 redis自封装了一个listNode用于承载链表节点，封装一个list用于存储头尾节点以及相关信息（长度等），但同时链表需要保存前后节点，内存开销变大，同时链表节点不连续，对元素是进行逐个访问的 压缩列表 压缩列表是一种内存紧凑类型的数据结构，占用一块连续的内存空间，其特点就是针对不同长度的数据，采用不同的编码，有效的节省内存开销 压缩列表保存较多数据的时候查询效率会降低，同时新增或者修改某个元素的话，压缩列表占用的内存空间就需要重新分配，这可能会造成连锁反应，导致后面的压缩列表一并重新分配内存，造成访问压缩列表性能的下降 所以对于压缩列表而言，适合保存数据量或者节点数量不太多的场景，这样进行连锁更新的成本也是可以接受的 跳表 有序链表一般只能够逐一查找元素，这样的操作非常缓慢，对于跳表来说，查询效率更高 IO网络模型

说一说IO网络模型 同步阻塞IO 首先对于同步阻塞io来说，从应用程序发起读取请求，到最终数据交付给进程，整个过程中进程是处于挂起的状态，当然这也有个优点：cpu可以处理其他进程，提高了进程的利用率。 等到内核数据缓冲区中数据准备好的时候，产生io中断通知内核数据已经准备好，内核响应中断，唤醒对应因为等待数据而挂起的进程，进程开始占用cpu时间将数据从内核缓冲区拷贝至用户态缓冲区中，最后进程来处理数据 **值得注意的是，从开始调用读取数据，再到开始处理数据，整个进程实际上是阻塞在这个地方的，**当然这种io模型对于高并发的状况来说的话是很不利的，这意味着假如1万个请求过来，结果全部阻塞在第一个请求后面，这样的性能就非常差了 同步非阻塞IO 和同步阻塞io有点类似，区别就是对cpu时间的占用方面有差别，进程通过自身用户态的循环询问，也就是循环进行系统调用，来查看内核数据缓冲区的数据是否准备好，当然这个时候进程同样可以做其他的事情，这个过程会是持续占用cpu运行时间的，所以循环的进行系统调用（不停的进行用户态和内核态的切换），降低了cpu的利用率，相比较于阻塞io的好处就是能在等待过程中处理进程的其他事件 IO多路复用 通过阻塞io和非阻塞io，我们发现阻塞io会造成并发请求的串行化处理，而非阻塞io则会造成cpu资源的浪费，于是诞生了io多路复用技术，多路复用技术其实是运用于通信方向的，多路信号同时复用同一个信道。网络部分也有运用，例如http2.0的多路复用技术 和非阻塞io不同的是，进程将需要监控的文件的文件描述符直接通过系统调用传给内核，直接通过内核来实现对所需数据是否准备好的监控，内核可以同时监控多个文件描述符，当文件描述符中有io读写操作时再通过内核进行返回通知对应进程，这也使得并发请求的高效处理成为了可能 select select调用通过传入一个含有文件描述符的集合，请求内核对对应的文件描述符进行监控，要注意的是，这个文件描述符集合是通过拷贝的方式传入的，会有一定的cpu资源损耗，同样，对于文件描述符集合中发生了io读写操作之后，内核返回给进程的同样是一个集合，相当于只通知进程有事件发生，但是不知道是哪一个，这就需要进程在用户态遍历集合找出准备好数据的文件描述符。对于少量的请求处理是可以的，但是对于像C10K这种问题而言，一次通知将遍历大量文件描述符 对于select而言，有着以下几个缺点： linux对select传入的文件描述符的大小有限制，一般是不超过1024个的，这也就是说能够处理的请求有限 对于文件描述符集合的拷贝部分是由用户空间拷贝至内核空间的，会有cpu资源的损耗 内核不能够精确通知进程哪一个文件描述符就绪，需要进程自己遍历集合 poll 相比较于select，取消了传入文件描述符的数量限制，但依旧没有解决其他两个问题 epoll 待更。。。
